## 1장. 깨끗한 코드

- 시간이 없어서 쓰레기 코드를 짜고 나중에 손보겠다고 생각한 경험이 다들 있겠지만, 나중은 결코 오지 않는다
- 여러 유명 프로그래머에게 물은 "깨끗한 코드란?"
  - 보기 즐거운 코드
  - 효율적인 코드
  - 나쁜 코드로 유혹하지 않는 코드
  - 철저한 오류처리
  - 한 가지만 잘 하는 코드
  - 잘 쓴 문장처럼 읽히는 코드
  - 다른 사람이 고치기 쉬운 코드
  - 테스트 케이스가 있는 코드
  - 시간 들여서 깔끔하고 단정하게 정리된 코드
  - 중복이 없는 코드
  - 작게 추상화한 코드
  - 짐작했던 기능을 그대로 수행하는 코드 (읽으면서 놀랄 일이 없는 코드)
  - 등등..

## 3장. 함수

- 작게, 더 작게!

- 한 가지만 제대로 해라

  - 추상화 수준이 하나인 단계만 수행한다? -> 그 함수는 한 가지 작업만 한다
  - 중간 로직을 의미 있는 이름으로 다른 함수로 추출할 수 있다? -> 그 함수는 여러 작업을 하는 중이다

- 함수 당 추상화 수준은 하나로

  - 한 함수 내에서 추상화 수준이 섞여있으면 읽는 사람이 이 코드가 근본 개념인지, 세부사항인지 헷갈려한다

- 코드는 위에서 아래로 이야기처럼

- switch문 말고 다형성을 이용하라

- 함수 이름은 서술적인 이름을 사용하라. 이름을 보고 짐작했던 기능을 그대로 수행한다면 좋은 코드.

- 함수 인수는 적을수록 좋다

- 부수 효과를 일으키지 마라. 메서드 명에 표현되지 않은 일은 하지 마라.

- 출력 인수는 자제하자

  ```java
  appendFooter(s); // s에 추가되는걸까? s를 추가하는걸까?
  public void appendFooter(StringBuffer report) { ... } // s에 추가되는 거구나..
  s.appendFooter() // 출력 인수는 자제하고 이렇게 쓰자
  ```

- 명령과 조회를 분리하자

  ```java
  // as-is
  public boolean set(String attribute, String value); // 속성의 값을 설정한 후 성공여부 반환
  if (set("username", "junho")) ...
  
  // to-be
  public boolean attributeExists(String attribute);
  public void setAttribute(String attribute, String value);
  if (attributeExists("username")) {
  	setAttribute("username", "junho");
    ...
  }
  ```

- 반복하지 마라. 반복은 악의 근원
